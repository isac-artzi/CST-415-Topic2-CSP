What is a Constraint Satisfaction Problem?

Simple Definition: A CSP is like solving a puzzle where you need to fill in blanks while following specific rules.

Think of it like organizing a seating chart for a wedding: you have guests (variables), available seats (domains), and rules like "keep feuding relatives apart" (constraints). You need to assign every guest to a seat while respecting all the rules.

The Three Components:

Variables

What: The "blanks" to fill in

X = {X₁, X₂, ..., Xₙ}

Example: Each guest at the wedding, each queen on the chessboard, each region on a map

Domains

What: The possible choices for each blank

D = {D₁, D₂, ..., Dₙ}

Example: Available seats, chess positions, colors

Constraints

What: The rules that must be followed

C = {C₁, C₂, ..., Cₘ}

Example: "Can't sit together," "can't attack," "can't be same color"

Goal: Find an assignment of values to variables that satisfies all constraints

Complete Assignment: Every variable has a value (all blanks filled)
Consistent Assignment: No constraints are violated (all rules followed)
Solution: Complete + Consistent (puzzle solved!)

Classic Example: Map Coloring

Real-World Analogy: Imagine you're creating a board game with different territories. You want neighboring territories to have different colors so players can see the borders clearly.

Problem:

Color a map of Australia so that no two adjacent regions share the same color

Breaking it Down:

  • What we're deciding: The color of each Australian state/territory
  • Available choices: Red, Green, or Blue for each region
  • Rules to follow: If two regions touch, they must be different colors

CSP Formulation:

  • Variables: WA, NT, SA, Q, NSW, V, T (7 regions)
  • Domains: {Red, Green, Blue} for each region
  • Constraints: Adjacent regions must have different colors
Example Constraints (in plain English and math):
• "Western Australia can't be the same color as Northern Territory" → WA ≠ NT
• "Western Australia can't be the same color as South Australia" → WA ≠ SA
• "Northern Territory can't be the same color as South Australia" → NT ≠ SA
• "Northern Territory can't be the same color as Queensland" → NT ≠ Q
• ... (and so on for all touching regions)

Why CSP? Without CSP, we'd try all 3⁷ = 2,187 possible combinations! CSP lets us eliminate bad choices early, often finding a solution after checking only 20-30 combinations.

Game Dev Connection: Use this for territory coloring, team assignments, resource allocation across game zones.

Game Example: N-Queens Problem

Game Context: Think of this like placing powerful enemy units in a strategy game - you want them spread out so they don't interfere with each other's attack patterns.

Problem:

Place N queens on an N×N chessboard so that no two queens attack each other

Why is this tricky? Queens in chess can attack any piece in the same row, column, or diagonal. We need to place them so none can "see" each other.

Smart Trick:

Instead of asking "where should I place all queens?", we simplify to: "I'll put one queen in each row - now which column should each go in?"

CSP Formulation:

  • Variables: Q₁, Q₂, Q₃, Q₄ (which column for the queen in each row)
  • Domains: {1, 2, 3, 4} (columns available) for each queen
  • Constraints:
    • No same column: If Q₁ = 2, then Q₂, Q₃, Q₄ can't be 2
    • No diagonal attacks: If queens are on rows i and j, their columns can't differ by |i - j|

In Math Terms:

  • No two queens in same column: Qᵢ ≠ Qⱼ
  • No two queens on same diagonal: |Qᵢ - Qⱼ| ≠ |i - j|

Lab Question: Knights on Chessboard

What Makes This Different from Queens? Knights have a unique L-shaped attack pattern (think of the "horse" piece in chess). They jump over pieces in an L-shape: 2 squares in one direction, then 1 square perpendicular.

Problem:

Place k knights on an n×n chessboard such that no two knights are attacking each other

Key Difference: Unlike queens, we don't get a shortcut! Each knight can be on ANY square, so this is actually harder to solve.

Breaking Down the Components:

  • What we're deciding: The exact position (row, column) for each knight
  • Available choices: Any of the n² squares on the board
  • Rules: No two knights on same square, and no L-shaped attacks

CSP Formulation:

Variables: K₁, K₂, ..., Kₖ (each knight's position on the board)

Domain: All squares on the n×n board
D = {(i,j) | 1 ≤ i,j ≤ n} — "Any row i and column j position"

Constraints: For any two knights Kᵢ and Kⱼ:
• Kᵢ ≠ Kⱼ — "Can't be on the same square"
• |xᵢ - xⱼ| + |yᵢ - yⱼ| ≠ 3 or |(xᵢ - xⱼ) × (yᵢ - yⱼ)| ≠ 2
  — "Can't be in L-shaped attack pattern"

Understanding the L-Shape: If Knight A is at (2,3), it attacks (1,1), (1,5), (3,1), (3,5), (4,2), (4,4), (0,2), (0,4). Notice each is exactly 2 away in one direction and 1 away in the other!

Game Application: Enemy spawn points that shouldn't overlap attack zones, tower placement in tower defense games.

Backtracking Search Algorithm

The Big Idea: Try to fill in values one at a time. If you hit a dead end (violate a constraint), undo your last choice and try something else. It's like solving a Sudoku puzzle!

How It Works (Plain English):

  1. Pick a variable that doesn't have a value yet (e.g., "Queen in row 2")
  2. Try a value from its domain (e.g., "put it in column 3")
  3. Check constraints - Does this break any rules?
  4. If OK: Move to the next variable and repeat
  5. If NOT OK: Try a different value
  6. If no values work: Go back to the previous variable and try a different value for it (this is the "backtracking")
  7. Success: When all variables have values and no rules are broken!

The Algorithm (Pseudocode):

function BACKTRACKING-SEARCH(csp) returns solution or failure
    return BACKTRACK({}, csp)

function BACKTRACK(assignment, csp) returns solution or failure
    if assignment is complete then return assignment

    var ← SELECT-UNASSIGNED-VARIABLE(csp)
    for each value in ORDER-DOMAIN-VALUES(var, assignment, csp) do
        if value is consistent with assignment then
            add {var = value} to assignment
            inferences ← INFERENCE(csp, var, value)
            if inferences ≠ failure then
                add inferences to assignment
                result ← BACKTRACK(assignment, csp)
                if result ≠ failure then return result
            remove {var = value} and inferences from assignment
    return failure

Why It's Called "Backtracking": When you hit a dead end, you "track back" to a previous choice and try a different path - just like exploring a maze!

Heuristic 1: Most Constrained Variable (MRV)

Also called: Minimum Remaining Values or the "fail-first" heuristic

The Question: When we have multiple empty variables to fill, which one should we tackle first?

The Answer: Pick the one with the FEWEST legal options remaining!

Why Does This Make Sense?

Imagine you're packing a suitcase with different-sized items:

  • If one item has only 2 spots it could fit, and another item has 10 possible spots...
  • Pack the constrained item first! If you can't fit it, you know immediately there's a problem
  • If you pack easy items first, you might pack the whole suitcase before discovering the hard item doesn't fit - wasted work!
Mathematical Definition:
MRV(X) = argmin |Domain(X)|

Translation: Choose the variable X with the smallest domain (fewest remaining values)

Why This Works:

  • Fail-first: If there's a problem, we discover it quickly (before wasting time on easier variables)
  • Pruning: Reduces the number of branches we explore
  • Efficiency: We avoid spending time on paths that will eventually fail

Sudoku Example:

Cell A can only be 3 or 7 (2 choices)

Cell B could be 1, 2, 3, 4, 5, 6, 7, 8, or 9 (9 choices)

MRV says: Fill Cell A first! If neither 3 nor 7 works, you know immediately. If you fill Cell B first, you might try many combinations before discovering Cell A has no solution.

Heuristic 2: Least Constraining Value (LCV)

The Question: Once we've picked a variable to fill (using MRV), which value from its domain should we try first?

The Answer: Pick the value that limits OTHER variables the LEAST!

Real-World Analogy:

You're scheduling meetings in conference rooms:

  • Meeting A could use Room 1 or Room 2
  • Meetings B, C, D can ONLY use Room 1
  • Smart choice: Put Meeting A in Room 2! This keeps Room 1 free for B, C, D
  • Bad choice: Put Meeting A in Room 1 - now B, C, D have no options!
Mathematical Definition:
LCV(v) = argmin Σ (constraints eliminated by assigning v)

Translation: Choose value v that removes the fewest options from other variables' domains

Combining MRV + LCV:

Variable Selection (MRV)

Most Constrained

"Tackle the hardest problem first"

✓ Fail fast - detect problems early

Value Selection (LCV)

Least Constraining

"Choose the most flexible option"

✓ Succeed quickly - preserve future options

Strategy: Pick the hardest variable first (MRV), but give it the easiest value (LCV)

Think: "Deal with difficult problems early, but handle them gently to keep future options open"

Inference: Forward Checking

The Idea: After making a choice, immediately update what's still possible for related variables. Don't wait to discover conflicts later!

Analogy: Planning a Road Trip

Imagine planning a route with multiple stops:

  • Without Forward Checking: "Let's visit City A at 2pm" → Later: "Oh no, we can't reach City B by 3pm from there!"
  • With Forward Checking: "If we visit City A at 2pm, then City B must be at 4pm or later (3pm is now impossible)" → Update immediately!

How It Works:

  1. Make an assignment (e.g., "Queen 1 goes in column 3")
  2. Look at all unassigned variables connected by constraints
  3. Remove values from their domains that would violate constraints
  4. If any variable ends up with NO valid values → we know this path fails!

The Algorithm:

function FORWARD-CHECKING(csp, var, value) returns inferences or failure
    inferences ← {}
    for each unassigned variable Y that is connected to var by constraint do
        for each value v in Domain(Y) do
            if v conflicts with value assigned to var then
                remove v from Domain(Y)
                add {Y ≠ v} to inferences
        if Domain(Y) is empty then return failure
    return inferences

Example: Map Coloring

Step 1: Assign WA = red

Step 2: Forward Checking updates neighbors:
• NT was {red, green, blue} → now {green, blue} (remove red)
• SA was {red, green, blue} → now {green, blue} (remove red)

Result: We immediately know NT and SA can't be red, saving us from trying those bad options later!

Benefit: Catch dead ends earlier, before wasting time exploring impossible paths.

Arc Consistency (AC-3 Algorithm)

The Big Idea: Forward Checking only looks one step ahead. Arc Consistency looks further, propagating constraints through the entire network!

What's an "Arc"?

Think of an arc as a one-way relationship between two variables: "X depends on Y"

  • Arc Consistency: Every value in X's domain has at least one compatible value in Y's domain
  • Analogy: Like a dating app - you only keep options where there's mutual compatibility!

Plain English Definition:

Arc (X → Y) is consistent if: For EVERY value that X could be, there's at least ONE value that Y could be that doesn't violate the constraint.

If not: Remove values from X's domain that have no compatible partner in Y!

Mathematical Definition:
Arc (X, Y) is consistent if:
∀x ∈ Domain(X), ∃y ∈ Domain(Y) such that constraint(X=x, Y=y) is satisfied

Translation: For all x in X's domain, there exists some y in Y's domain that works together

The AC-3 Algorithm:

Strategy: Keep revising domains until nothing changes anymore (we've found all the incompatibilities)

function AC-3(csp) returns false if inconsistency found, true otherwise
    queue ← all arcs in csp
    while queue is not empty do
        (X, Y) ← REMOVE-FIRST(queue)
        if REVISE(csp, X, Y) then
            if size of Domain(X) = 0 then return false
            for each Z in neighbors(X) except Y do
                add (Z, X) to queue
    return true

function REVISE(csp, X, Y) returns true iff Domain(X) was revised
    revised ← false
    for each x in Domain(X) do
        if no value y in Domain(Y) satisfies constraint(X,Y) then
            delete x from Domain(X)
            revised ← true
    return revised

Power: Often solves easy CSPs without any search! For hard CSPs, it dramatically reduces the domains before we start backtracking.

Detailed Example: Dungeon Room Placement

Problem: Generate a dungeon with 10 rooms that feels balanced and fun

CSP Setup:

  • Variables: R₁ (start), R₂...R₉ (normal), R₁₀ (boss)
  • Domains: Grid positions (x, y) on a 20×20 grid
  • Constraints:
    • Rooms must not overlap (no two rooms at same position)
    • Rooms must be reachable (connected by corridors)
    • Boss room must be far from start (distance ≥ 15)
    • Treasure rooms must be in dead ends (only 1 connection)
    • Minimum spacing between rooms (for corridors)

Result: CSP ensures every generated dungeon is playable and interesting, without manual design!

Summary & Key Takeaways

The Big Picture:

CSP is about systematically solving puzzles with rules. Instead of trying every possibility, we use smart strategies to find solutions efficiently.

What We Learned:

1. CSP Basics

Variables: What to decide

Domains: Possible choices

Constraints: Rules to follow

2. Backtracking

Try values one at a time

If stuck, undo and try different path

Like solving a maze!

3. Smart Ordering

MRV: Tackle hard problems first

LCV: Keep options open

Work smart, not hard!

4. Look Ahead

Forward Checking: Update immediately

AC-3: Propagate everywhere

Catch problems early!

Performance:
Worst case: O(d^n) — try all d^n combinations
With heuristics: Often dramatically better (exponential → polynomial)

Example: 10 variables, 10 values each
Brute force: 10^10 = 10 billion combinations
With CSP techniques: Often just thousands!

Your Implementation Checklist:

  1. Identify the CSP: What are you assigning? What are the rules?
  2. Formulate clearly: Write down variables, domains, constraints
  3. Implement backtracking: Basic search algorithm
  4. Add MRV heuristic: Choose hardest variable first
  5. Add LCV heuristic: Choose most flexible value
  6. Add inference: Forward checking or AC-3
  7. Test & measure: Count backtracks, time to solution
